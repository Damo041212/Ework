## 一、为什么要用 Git？

想象一下：

> 你正在写课程设计报告，改了很多版本，突然发现上周的版本更好，但是已经覆盖了...
> 
> 或者，你的电脑突然坏了，课程设计文件全丢了...
> 
> 又或者，小组合作时，同学改了你的文件，导致内容冲突...

**Git 就是来解决这些问题的！**

它就像一个：
- ✅ **自动保存机**：每一次修改都自动记录
- ✅ **时光穿梭机**：随时回到之前的版本
- ✅ **云端保险箱**：文件备份到网上，不怕丢失
- ✅ **协作神器**：小组合作时不会互相覆盖

## 二、准备工作（只做一次）

### 第一步：确认 Git 已安装

打开电脑的终端（命令提示符），输入：

```
git --version
```

如果显示版本号（比如 `git version 2.xx.x`），说明已安装。

### 第二步：告诉 Git 你是谁

```
git config --global user.name "你的名字"
git config --global user.email "你的邮箱"
```

这就像在图书馆借书时要登记名字一样，方便老师查看你的提交记录。

### 第三步：推荐工具（可选但强烈建议）

如果你觉得输入命令太麻烦，可以用图形界面工具，**点鼠标就可以操作**：

| 工具 | 特点 | 适合场景 |
|------|------|----------|
| **GitHub Desktop** | 专门为 Git 设计的软件，界面简洁 | 新手首选，纯图形操作 |
| **VS Code** | 代码编辑器自带 Git 功能 | 写代码时直接用 |

**推荐使用 GitHub Desktop**，下载地址：https://desktop.github.com/

安装后，所有操作都可以通过点击按钮完成，不用记命令！

## 三、常用操作（按顺序来）

### 操作 1：创建「档案柜」（初始化）

当你新建课程设计文件夹后，在文件夹里打开终端，输入：

```
git init
```

**效果**：在你的文件夹里创建了一个隐形的「档案柜」，从此以后所有修改都会被记录。

### 操作 1.5：创建「忽略清单」（.gitignore）

这一步非常重要！有些文件不需要保存到 Git（比如仿真生成的临时文件、大文件等），我们需要告诉 Git 忽略它们。

我已经为你创建了一个 `.gitignore` 文件，它会自动忽略：
- 仿真软件生成的临时文件（比如 MATLAB/Simulink 的 .mat、.slxc 文件）
- 系统自动生成的临时文件
- 大的输出文件

你不需要做任何操作，这个文件已经放在你的项目文件夹里了。

---

### 操作 2：放文件进档案柜（暂存）

每次写完文件后，先告诉 Git "我要保存这些文件"：

```
git add .
```

**效果**：把当前文件夹里所有修改过的文件，都放进档案柜的「待保存区」。

---

### 操作 3：锁上档案柜（提交）

放好文件后，给这次保存起个名字，然后锁上：

```
git commit -m "今天做了什么"
```

**例子**：
```
git commit -m "完成主电路仿真图"
git commit -m "修改控制参数，仿真结果更稳定"
git commit -m "添加课程设计报告第一章"
```

**效果**：给当前状态拍了一张快照，永远不会丢失。

---

### 操作 4：查看修改记录（历史）

想看看以前做了什么，输入：

```
git log
```

**效果**：显示所有提交记录，包括时间、内容。

---

### 操作 5：回到以前的版本（回退）

如果改坏了，想回到之前某个版本：

1. 先查看历史记录：`git log`
2. 找到想回到的版本号（一串字母数字）
3. 输入：`git checkout 版本号`

**例子**：
```
git checkout a1b2c3d
```

**效果**：回到那个版本的状态，就像时光倒流。

---

### 操作 6：开个「分柜」（分支）

想尝试新方案，但又不想破坏现有内容：

```
git branch 新方案名字
```

**例子**：
```
git branch PID控制方案
git branch 模糊控制方案
```

然后切换到新分支：
```
git checkout PID控制方案
```

**效果**：在不影响主文件的情况下，独立开发新方案。

---

### 操作 7：把分柜内容合回来（合并）

新方案测试通过后，合并回主分支：

```
git checkout main        # 先回到主分支
git merge PID控制方案    # 把分支内容合并过来
```

**效果**：两种方案的内容合并到一起。

---

### 操作 8：备份到云端（远程仓库）

为了防止电脑坏了丢失文件，上传到网上：

1. 先在 GitHub/Gitee 上注册账号，创建一个「仓库」
2. 按网站提示，输入：

```
git remote add origin https://你的仓库地址
git push -u origin main
```

**效果**：文件备份到网上，随时可以下载。

---

### 操作 9：从云端下载最新版本（拉取）

如果同学更新了文件，或者换了电脑，从云端同步：

```
git pull origin main
```

**效果**：下载最新版本的文件到本地。

## 四、课程设计实际场景示例

### 场景 1：日常开发流程

```
今天写了主电路仿真代码

1. git add .                    # 放进档案柜
2. git commit -m "完成主电路仿真"  # 锁上档案柜
3. git push origin main         # 备份到云端
```

### 场景 2：尝试不同方案

```
想试试 PID 和模糊控制两种算法

1. git branch PID方案           # 开个分柜
2. git checkout PID方案         # 切换到分柜
3. 写 PID 控制代码...
4. git add . && git commit -m "完成PID方案"
5. git checkout main            # 回到主柜
6. git branch 模糊控制方案       # 再开个分柜
7. git checkout 模糊控制方案     # 切换过去
8. 写模糊控制代码...
```

### 场景 3：改坏了要撤回

```
不小心删掉了重要代码

1. git log                      # 查看历史记录
2. 找到删除前的版本号（比如 a1b2c3d）
3. git checkout a1b2c3d         # 回到那个版本
4. 文件恢复了！
```

### 场景 4：小组合作

```
和同学一起做课程设计

1. 同学上传了新文档 → 你输入：git pull origin main
2. 你修改了代码 → git add . → git commit -m "修改了XX"
3. 上传你的修改 → git push origin main
4. 同学再拉取 → git pull origin main
```

### 场景 5：标记最终版本

```
完成答辩版本，要提交给老师

1. git tag -a "答辩版本" -m "2026年7月5日提交老师"
2. git push origin "答辩版本"
```

**效果**：以后可以直接找到提交给老师的那个版本。

## 五、常见问题解决

| 问题 | 解决方法 |
|------|----------|
| 输入命令后显示红色错误 | 复制错误信息，问同学或老师 |
| 提示文件未追踪（Untracked） | 先执行 `git add .` |
| 提交后想修改备注 | 用 `git commit --amend -m "新备注"` |
| 忘记密码 | 在 GitHub/Gitee 上重置密码 |
| 不知道当前状态 | 输入 `git status` 查看 |

## 七、重要提醒

1. **每天结束前要做的事**：`git add .` → `git commit -m "今天做了什么"` → `git push origin main`
2. **重要版本一定要提交**：完成一个模块、修改关键参数后，都要提交
3. **提交备注要写清楚**：比如"修改了滤波电容参数"，不要写"改了一下"
4. **不怕出错**：Git 可以撤回几乎所有操作，大胆尝试
